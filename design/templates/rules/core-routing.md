# 输入路由规则

每轮输入先做语义分类，再映射到对应处理方式。
**分类是语义判断，不是关键词硬匹配**；有歧义时选覆盖范围更广的分类。

---

## 分类表

| 分类 | 意图特征 / 关键词 | 处理方式 |
|---|---|---|
| 闲聊 / 寒暄 | 问候、感谢、随便聊、「你好」「今天怎么样」 | 直接回答，不写文件，不触发 skill |
| 一次性问答 | 单问题、概念解释、「XX 是什么」、状态查询 | 直接回答（需要时检索 KB），不写文件 |
| 搜索 / 调研 | 「帮我找」「查一下」「有哪些」「调研」「搜一下」 | 检索 KB，整理后回答或落文件（用户要求保存时） |
| 设计 / 方案 / 规划 | 「怎么设计」「架构」「方案」「计划」「PRD」「规划」 | 讨论后产设计文档到 outputs/ |
| 内容生成 | 「出一篇」「生成内容」「做个图文」「写小红书」「朋友圈文案」「家长群话术」「微信群话术」「社群话术」「书单」「读书笔记」「知识卡片」「读后感」「书评」「推荐语」「配图」「短视频」「视频脚本」 | 触发 content-generate skill |
| 执行 / 任务 / 代码 | 「实现」「写代码」「修 bug」「写脚本」「提交」「改配置」 | 执行模式，改文件，git commit（用户确认后） |

---

## 路由优先级规则

1. **「内容生成」优先于「搜索」**：含「生成/出/做/写+成品形态（图文/视频/文案/书单等）」的意图直接走 content-generate，不走搜索
2. **「一次性问答」 vs「搜索」**：「解释 X / X 是什么」→ 一次性问答；「帮我找 X 的资料 / 有哪些 X」→ 搜索
3. **每轮只触发一个主处理方式**；复合需求（先调研再生成）分步处理，每步分开分类
4. **不确定时**：向用户澄清意图，不自行猜测

## 默认行为（未命中 skill 的分类）

### 闲聊 / 寒暄
直接回答，简短，语气自然，不写文件。

### 一次性问答
1. 优先从已加载 rules 和 memory/summary.md 中回答
2. 需要查 KB 时：`python skills/content-generate/scripts/content_runtime.py kb search --query "<关键词>" --modality doc --topk 3`
3. 回读命中文档后回答，不写文件

### 搜索 / 调研
1. 解析搜索意图（主题 / modality 偏好 / 数量）
2. `python skills/content-generate/scripts/content_runtime.py kb search --query "<主题>" --modality all --topk 10 --json`
3. 回读前 3-5 条命中文件
4. 整理为列表回答（附 source_path 和相关度说明）
5. 用户要求「保存结果」时写入 `outputs/YYYY-MM-DD/research/<topic>.md`

### 设计 / 方案 / 规划
1. 先讨论：目标/约束/方案对比（不急着写文件）
2. 用户确认方向后，产设计文档到 `outputs/YYYY-MM-DD/design/<title>.md`
3. 写文件前展示拟写内容，等用户确认
4. 触发 finalize

### 执行 / 任务 / 代码
1. 分析任务，制定计划
2. 执行代码/文件变更
3. 验证（语法检查/测试/启动确认）
4. 展示拟提交内容，等用户确认后 `git add <files> && git commit -m "<message>"`
5. 触发 finalize
